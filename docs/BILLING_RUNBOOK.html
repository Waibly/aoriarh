1. Architecture & concepts

AORIA RH distingue deux catégories de plans qui cohabitent sur le champ Account.plan :

CatégorieValeursFacturationAssignation
Techniques internes gratuit, invite, vip Hors Stripe Manuel via back-office (ou automatique pour gratuit = essai 14 j)
Commerciaux Stripe solo, equipe, groupe Stripe Checkout + webhook Self-service depuis /billing

Statut du compte (Account.status) indépendant du plan :

  • active — accès complet
  • trialing — essai en cours
  • past_due — 1er échec de paiement, accès maintenu
  • suspended — 2 échecs ou essai expiré, accès bloqué
  • canceled — résiliation volontaire, accès fermé

Source de vérité des limites : le fichier backend/app/core/plans.py (dataclass PlanLimits). Modifier un chiffre = modifier du code + déploiement. Aucune migration DB requise.

Politique fair-use

Le quota de questions mensuel est partagé par compte (pot commun entre tous les users et orgs). Le dépassement ne bloque jamais : seuils d'alerte :

  • Soft warning ≥ 80 % → bannière orange in-app
  • Hard warning ≥ 120 % → bannière rouge + incitation à upgrade

Les seuils sont dans backend/app/schemas/billing.py (SOFT_WARNING_RATIO / HARD_WARNING_RATIO).

Cascades de suppression

Toutes les FK vers accounts.id sont en ON DELETE CASCADE (subscriptions, addons, boosters, monthly_usage). La suppression d'un Account nettoie automatiquement ces tables. Les organisations / documents / conversations ne sont pas en cascade — ils doivent être supprimés explicitement (voir DataRetentionService.purge_account).

6. Jobs planifiés

Gérés par arq dans backend/app/worker.py.

TâcheFréquence (UTC)Rôle
run_billing_lifecycle Tous les jours à 9 h Envoie les emails d'essai J-7 et J-1 (J-3 volontairement omis pour éviter la perception de spam). À J+0, marque l'essai suspended + envoie l'email d'expiration. Révoque aussi les plans invite expirés.
run_data_retention_purge Tous les jours à 10 h Supprime définitivement les comptes ayant dépassé leur fenêtre de rétention (30 j après essai/résiliation, 60 j après impayé). Nettoie PostgreSQL + MinIO + Qdrant.
run_scheduled_sync 1er et 15 du mois à 3 h Existait déjà — sync jurisprudence / CCN / codes juridiques. Indépendant du billing.

Le tick quotidien suffit pour la cadence business (trial 14 j). Chaque email n'est envoyé qu'une seule fois : la fenêtre de match (6 ≤ remaining < 7 jours pour J-7, etc.) est exacte sur 24 h.

7. Flows utilisateur

Signup → essai → upgrade

J0   signup /register
     ├─ User + Account créés via AuthService._new_trial_account()
     ├─ plan='gratuit', status='trialing'
     ├─ plan_assigned_at = now(), plan_expires_at = now() + 14 jours
     └─ Access complet (limites Essai : 1/1/20/150q)

J+1..J+13  utilisation normale
     ├─ Chaque question → BillingService.check_question_quota()
     │    + increment_question_count() après succès
     └─ Banner /dashboard affiche le countdown trial

J+7   email "Votre essai se termine dans 7 jours"
J+13  email "... demain"

J+14  cron lifecycle détecte remaining < 0
     ├─ Account.status ← 'suspended'
     ├─ Email "Votre essai est terminé"
     └─ Tous endpoints protégés renvoient 402 Payment Required

Opt 1 : user clique "Choisir une offre" → POST /billing/checkout
     ├─ Stripe Customer créé (get_or_create)
     ├─ Session Checkout créée avec plan + cycle
     ├─ Redirect vers Stripe
     ├─ Paiement → webhook checkout.session.completed
     │    ├─ Subscription créée
     │    ├─ Account.plan ← plan commercial
     │    └─ Account.status ← 'active'
     └─ Redirect /settings/billing?success=1

Opt 2 : user ne souscrit pas
     └─ J+44 (14 + 30)  cron purge → suppression définitive

Dépassement de quota

Les questions ne sont jamais bloquées. Le statut renvoyé par BillingService.check_question_quota() déclenche :

  • soft_warning (≥ 80 %) : bannière orange, lien vers /billing
  • hard_warning (≥ 120 %) : bannière rouge, proposition booster ou upgrade

Le booster (+500 q à 25 €) est consommé en FIFO après l'épuisement du quota mensuel.

Cas d'impayé

Stripe retente 3-4 fois automatiquement. Chaque échec :

  • 1er échec : invoice.payment_failed → past_due, accès maintenu
  • 2e échec (attempt_count ≥ 2) : suspended, accès bloqué
  • invoice.paid ultérieur : retour à active
  • Après 60 j de suspension sans régularisation : purge cron

9. Opérations courantes

Bootstrap / mise à jour du catalogue Stripe

docker compose exec -w /app backend python -m scripts.stripe_bootstrap

Idempotent : relance sans risque. Affiche le bloc STRIPE_PRICE_* à coller dans .env.

Déclencher manuellement le cron lifecycle

docker compose exec -w /app backend python -c "
import asyncio
from app.worker import run_billing_lifecycle, on_startup

async def main():
    ctx = {}
    await on_startup(ctx)
    await run_billing_lifecycle(ctx)
    await ctx['engine'].dispose()

asyncio.run(main())
"

Déclencher manuellement la purge RGPD

docker compose exec -w /app backend python -c "
import asyncio
from app.worker import run_data_retention_purge, on_startup

async def main():
    ctx = {}
    await on_startup(ctx)
    await run_data_retention_purge(ctx)
    await ctx['engine'].dispose()

asyncio.run(main())
"

Assigner manuellement un plan (admin)

Depuis le backoffice : /admin/users → sélectionner un user → bouton "Changer de plan" (vip permanent / invite X mois).

Répondre à une demande RGPD

  1. Export art. 15/20 : GET /api/v1/admin/accounts/{id}/export → fichier JSON à envoyer au demandeur.
  2. Effacement art. 17 : POST /api/v1/admin/accounts/{id}/erase → supprime immédiatement.
  3. UI : /admin/billing → table "Comptes à purger" → boutons Export / Supprimer.

Surveiller les KPIs

Page /admin/billing : MRR, ARR, trials actifs, churn 30 j, répartition par plan. Rafraîchie à chaque chargement.

11. RGPD & rétention

Fenêtres de rétention

SituationAncre temporelleGrâce
Essai expiré non convertiplan_expires_at (exact)30 jours
Résiliation volontaireupdated_at (approximation)30 jours
Impayé / suspension Stripeupdated_at (approximation)60 jours

Effet de la purge

Par ordre, pour chaque compte éligible :

  1. Vecteurs Qdrant → suppression par FilterSelector(organisation_id)
  2. Fichiers MinIO → StorageService.delete_file(storage_path) pour chaque Document
  3. PostgreSQL : messages → conversations → documents → conventions → memberships → organisations → invitations → account_members → account (cascade sur subscriptions/addons/boosters/usage) → user owner

Irréversible : aucune corbeille, aucun backup auto. Si vous devez préserver une trace (ex : obligations comptables Stripe), exporter d'abord via GET /admin/accounts/{id}/export.

Données conservées côté Stripe

Les factures, reçus et historiques de paiement restent sur Stripe (cf. obligations comptables 10 ans). La purge AORIA ne supprime pas le Customer Stripe — il reste orphelin dans leur système.

12. Roadmap (non implémenté)

  • Zero Data Retention OpenAI — à activer côté OpenAI entreprise (demande support). Mentionné dans la politique de confidentialité.
  • Anonymisation PII avant envoi LLM — sujet important, reste à spécifier (Presidio / regex NIR-BIC).
  • Chiffrement au repos (PostgreSQL / Qdrant / MinIO) — prévu, mention honnête dans la politique.
  • Sauvegardes chiffrées externalisées — hors du VPS principal.
  • Champ Account.status_changed_at — pour ancrer précisément la fenêtre de rétention sur les suspensions / résiliations (aujourd'hui updated_at).
  • Add-ons commandés depuis l'UI — les endpoints Stripe sont prêts mais il manque l'UI pour ajouter / retirer des add-ons depuis /billing (actuellement via Stripe Customer Portal uniquement).
  • Dunning emails custom — aujourd'hui Stripe envoie ses propres mails de relance. On peut surcharger avec nos templates Brevo.
  • Dashboard analytics historiques — évolution MRR / churn dans le temps, cohortes (actuellement seulement un snapshot).
  • Tests d'intégration automatisés — tests E2E sur le flow signup → trial → checkout → webhook → purge.
